# 초보자를 위한 GitHub Pages 배포 가이드

이 문서는 우리가 만든 웹사이트를 전 세계 사람들이 볼 수 있도록 **인터넷에 올리는 과정(배포)**을 설명합니다. 코딩을 몰라도 이해할 수 있도록 핵심 개념과 과정을 풀어서 정리했습니다.

---

## 1. 핵심 개념 (이것만 알아도 절반은 성공!)

### 🌍 배포 (Deployment)
내 컴퓨터(Local)에 있는 코드를 실제 사용자들이 접속할 수 있는 **서버(Server)**로 옮기는 작업입니다. 우리는 **GitHub Pages**라는 무료 서버를 사용했습니다.

### 🤖 CI/CD (지속적 통합/배포)
- **CI (Integration)**: 코드를 합칠 때 문제가 없는지 자동으로 검사하는 것.
- **CD (Delivery/Deployment)**: 검사가 끝난 코드를 자동으로 배포하는 것.
- **우리가 한 일**: "코드를 GitHub에 올리면(Push), 자동으로 사이트를 만들어서(Build), 배포해라"라고 로봇에게 시킨 것입니다.

### ⚙️ 빌드 (Build)
우리가 작성한 코드(React, 최신 자바스크립트)는 브라우저가 바로 이해하기 어렵습니다. 이를 브라우저가 이해할 수 있는 **최적화된 파일(HTML, CSS, JS)**로 변환하는 과정을 '빌드'라고 합니다. 요리에 비유하면 '재료 손질 및 조리' 과정입니다.

---

## 2. 우리가 수행한 배포 과정 3단계

### 1단계: 주소지 설정 (`vite.config.js`)
택배를 보낼 때 주소가 정확해야 하듯이, 우리 사이트가 어디에 위치할지 알려줘야 합니다.
- **문제**: 기본적으로 사이트는 `도메인.com/` 경로를 예상합니다.
- **해결**: 우리는 `항글라이터.github.io/geminipromptc/`라는 하위 주소를 쓰므로, `/geminipromptc/`라는 **기본 경로(Base Path)**를 설정해주었습니다.

### 2단계: 자동화 로봇 고용 (`deploy.yml`)
매번 수동으로 빌드하고 파일을 올리는 건 귀찮습니다. GitHub Actions라는 로봇에게 작업 지시서를 주었습니다.
- **지시서 위치**: `.github/workflows/deploy.yml`
- **지시 내용**:
  1. "누가 `main` 브랜치에 코드를 올리면 깨어나라."
  2. "Ubuntu(리눅스) 컴퓨터를 하나 빌려라."
  3. "Node.js를 설치하고, 필요한 프로그램(`npm ci`)을 깔아라."
  4. "사이트를 만들어라(`npm run build`)."
  5. "만들어진 파일을 GitHub Pages 서버에 올려라."

### 3단계: 실행 및 확인
- **Git Push**: 코드를 저장소에 올리는 순간, 위에서 설정한 로봇이 자동으로 일을 시작합니다.
- **문제 해결**: 처음에 로봇이 쓰는 도구(Actions)가 너무 낡았다는 경고가 떠서, 최신 버전(`v4`)으로 업그레이드해주었습니다.

## 3. 자주 쓰는 용어 사전 (Glossary)

| 용어 | 설명 | 비유 |
|------|------|------|
| **Repository (레포)** | 프로젝트 파일이 저장된 공간 | 프로젝트 폴더 |
| **Branch (브랜치)** | 코드의 복사본. 여러 버전 작업을 할 때 사용 | 평행 우주 |
| **Commit (커밋)** | 작업한 내용을 저장하고 기록을 남기는 것 | 게임의 세이브 포인트 |
| **Push (푸시)** | 내 컴퓨터의 커밋을 GitHub(원격 저장소)로 보내는 것 | 클라우드 업로드 |
| **Action (액션)** | GitHub에서 특정 이벤트(푸시 등)가 발생하면 실행되는 작업 | 자동화 매크로 |

---

## 🎯 결론
이제 여러분은 코드를 수정하고 **Push**만 하면, 복잡한 과정 없이 자동으로 사이트가 업데이트됩니다. 이것이 현대적인 개발의 **자동화 배포** 시스템입니다!
